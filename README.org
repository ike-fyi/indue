* Introduction

Indue is a markup language designed from scratch for outliner, task management,
and personal knowledge management (PKM) use cases. It draws inspiration from
Typst's syntactic sensibility but is not a subset, superset, or dialect of
Typst — it is an independent language that could transpile to Typst for typeset
export when needed.

The name "Indue" comes from the archaic English verb meaning "to put on as a
garment" — to clothe or endow something with qualities it did not previously
have. Plain text puts on Indue and gains structure, tasks, tags, references,
and hierarchy. Remove the markup and the text is still there, perfectly
readable. This metaphor of equipping text with power runs through the entire
language and its vocabulary.

Indue files are plain text. They are readable and writable in any editor
without special tooling. The markup is lightweight enough that it does not
obscure the content it describes.

* Inspirations

** Typst

Typst is a modern typesetting language with a clean, consistent syntax. Indue
borrows directly from it:

- Bold (~*text*~) and the general aesthetic of lightweight paired delimiters.
- List markers (~-~ for unordered, ~+~ for ordered).
- Heading syntax (~=~ / ~==~ / ~===~).
- Labels (~<label>~) and references (~@ref~).
- The principle that functions and evaluation should have a clear, single
  prefix character.

Where Indue parts ways with Typst:

- Typst reserves ~#~ for function calls. Tags — one of the most natural and
  important constructs in PKM — require a function call like ~#tag("work")~.
  Indue reclaims ~#~ for native tag syntax and moves functions to ~:~.
- Typst has no built-in concept of tasks, checkboxes, or task states. These
  must be added through packages and show rules. Indue makes them native.
- Typst is a strict compiler: malformed input is an error. Indue follows an
  HTML-style error recovery model where malformed markup renders as literal
  text rather than failing.

** Org-mode

Org-mode is the most powerful plain-text outliner and task system in existence,
but its spec and runtime are so entangled with Emacs that every non-Emacs
implementation is permanently incomplete. Neovim and Helix users are
permanently second-class citizens.

Indue borrows the /idea/ that tasks, scheduling, and metadata belong in the
markup itself. It does not inherit Org's property drawers, babel blocks, or
export dispatcher. More importantly, Indue is deliberately designed to be fully
and correctly implementable by anyone — a Rust reference CLI, a tree-sitter
grammar, and a clean formal spec mean that any editor gets first-class support
from day one, not as an afterthought.

** CommonMark

Indue is not trying to be CommonMark. CommonMark is purely presentational with
no opinion about what content /means/. Indue is positioned between CommonMark
and Org-mode: richer than CommonMark in semantic content (tasks, tags, entities,
scheduling), simpler and more portable than Org-mode, and without Org's
assumption of a specific runtime.

** Everyday conventions

- ~#~ as tag comes from social media and is universally understood.
- ~:~ as function/evaluation is lightweight and familiar from emoji shortcodes,
  YAML keys, and general programming conventions.

* Vocabulary

Indue uses a thematic vocabulary inspired by RPG and medieval conventions to
name its syntactic concepts. These names are optional — the common markup
names are always valid and appear alongside the Indue terms throughout this
specification. The thematic vocabulary exists to make the concepts memorable,
teachable, and distinct from other markup languages.

The core metaphor: plain text is your base character. Indue is the gear system.
You equip your text with markup and it gains new abilities. Strip the gear off
and the character is still there, still whole.

| Syntax                   | Indue name      | Common name          |
|--------------------------+-----------------+----------------------|
| ~-~, ~+~, ~=~           | *Socket*        | Structural marker    |
| Node created             | *Slot*          | Block / node         |
| — heading slot           |                 | Foldable, in ToC     |
| — list slot              |                 | Not foldable, leaf   |
| ~*~, ~/~, ~_~, ~%~, ~^~ | *Glamour*       | Inline formatting    |
| ~#~                      | *Badge*         | Tag                  |
| ~@~                      | *Signet*        | Reference / mention  |
| ~:fn~                    | *Spell*         | Function             |
| ~::~                     | *Cast*          | Scope / chain op     |
| ~()~                     | *Vessel*        | Value argument       |
| ~[]~                     | *Scroll*        | Verbatim block       |
| ~{}~                     | *Conduit*       | Rich content block   |
| ~<label>~                | *Waypoint*      | Label / anchor       |
| ~<<id>>~                 | *Coordinate*    | Block ID             |
| ~\~                      | *Nick*          | Escape character     |
| ~:::~                    | *Cauldron*      | Spell block scope    |
| ~\~~                     | *Satchel*       | Hoisted config block |
| Config block             | *Codex*         | Inherited properties |

A one-sentence summary of the entire system: sockets create slots; glamours,
badges, and spells equip the content sitting on those slots; casts direct
spells at their targets and chain spells together; and vessels, scrolls, and
conduits carry data between them. Codexes configure how glamours, sockets,
and entities bestow their properties.

* Core Design Philosophy

** Prefer unambiguous syntax

Indue prefers syntax where every character's role is resolvable from local
context. The parser should never need to look ahead through unbounded content
or backtrack to determine meaning. Characters may appear in multiple roles
(e.g. ~:~ opens a spell, terminates one, and participates in casts), but
context always disambiguates which role applies.

** Glamours are for frequent writing; spells are for transforms

The bar for earning a dedicated glamour is high: the feature must appear
frequently enough in everyday writing that the ergonomic cost of a spell
(function call) is genuinely burdensome. Anything that transforms content into
a different form belongs in the spell system (~:func~). The glamour table is
for things you type constantly.

** The line-start rule

Any inline glamour appearing at the start of a line, followed by a space and
without a closing delimiter, applies to the entire line. Multiple glamour
characters can be stacked at line start:

#+begin_example
* This whole line is bold
/ This whole line is italic
*/ This whole line is bold and italic
_% This whole line is underlined and struck through
# taggedline
#+end_example

This is especially useful inside lists where content length may change:

#+begin_example
- = Project Alpha
  - * This important item is bold regardless of how long it gets
  - */ This one is bold and italic
  - Normal item
#+end_example

** The single-word rule

A glamour touching the left side of a word (no space between glamour and
first character) opens markup that terminates at the next whitespace. This
is the same logic used by badges (~#tag~) and signets (~@alice~):

#+begin_example
This is *bold right here         — "bold" is bold, rest is normal
She has /style in spades         — "style" is italic
#+end_example

To use a glamour character literally in this position, escape it with a
nick: ~\*minute~ renders as a literal asterisk.

** The paired rule

If a matching closer exists on the same line, the glamour spans multiple
words:

#+begin_example
This has *bold* in it           — paired, one word
This has *bold words* in it     — paired, multi-word
This is *bold right here        — no closer: single-word rule applies
#+end_example

The parser scans the current line for a closer. Found one? Paired mode.
No closer? Single-word, terminate at whitespace.

** The bracket rule

Glamours can also scope their content using scrolls ~[]~ or conduits ~{}~:

#+begin_example
*[bold words]               — scroll defines scope
*{bold with #tags inside}   — conduit: inner content is parsed
#+end_example

This is the only way to combine multiple glamours (stacking):

#+begin_example
*/[bold and italic]         — glamour stack + scroll
_*{underline and bold}      — glamour stack + conduit
%/[strikethrough and italic] — glamour stack + scroll
#+end_example

Adjacent glamour characters with no whitespace form a stack. The bracket
immediately after the stack defines the scope. Order of glamours in the
stack does not matter.

** No symmetric nesting

Glamours cannot be nested by wrapping them symmetrically. The outer glamour
wins and inner glamour characters are treated as literal text:

#+begin_example
*_text_*     → just bold. Inner _ is literal.
_*text*_     → just underline. Inner * is literal.
#+end_example

To combine glamours, always use the bracket form: ~*_[text]~.

** The nick (~\~)

~\~ is reserved as an escape character. Its primary use is rendering a
literal glamour character that would otherwise be interpreted as markup:

#+begin_example
The price is 50\% off       — literal %, not strikethrough
Use \*asterisks\* literally — literal *, not bold
#+end_example

~\~ before a non-glamour character renders as a literal backslash.

** Parse, don't fail

Indue parsers should never throw errors at the user for malformed markup.
Unclosed glamour delimiters apply to the single word they touch (single-word
rule), or render as literal text if not touching a word. Unknown spell names
render their content as plain text. The no-errors philosophy follows HTML's
error recovery model rather than Typst's strict compilation model.

Strict validation belongs in the linter and export pipeline, not in the core
parser that runs while someone is writing notes at midnight.

* Formatting Spells

Indue provides standard formatting spells that work with any text content.
These spells follow all standard cast and chaining rules.

** Common formatting spells

| Spell                | Effect                                  |
|----------------------+-----------------------------------------|
| ~:bold~              | Bold (spell form of ~*~)                |
| ~:italic~            | Italic (spell form of ~/~)              |
| ~:underline~         | Underline (spell form of ~_~)           |
| ~:strike~            | Strikethrough (spell form of ~%~)       |
| ~:super~             | Superscript (spell form of ~^~)         |
| ~:color(value)~      | Text color (named or hex)               |
| ~:bgcolor(value)~    | Background/highlight color              |
| ~:bgradius(value)~   | Highlight shape (rounded, pill, etc.)   |
| ~:font(type)[name]~  | Font (vessel=fallback type, scroll=name)|
| ~:grow(n)~           | Increase font size by n steps           |
| ~:shrink(n)~         | Decrease font size by n steps           |
| ~:mono~              | Monospace font                          |
| ~:opacity(value)~    | Transparency (0.0 to 1.0)              |
| ~:align(value)~      | Text alignment (left, center, right)    |

These chain naturally with glamours and each other:

#+begin_example
:bold::color(red)::word             — bold + red, one word
:italic::grow(1):: whole line       — italic + larger, line scope
:color(blue)::bgcolor(#ffffcc)::[highlighted text]  — colored + highlighted
#+end_example

** Sigil configuration (~:char~)

Every glamour and socket character has a corresponding configuration spell.
The configuration spell uses the literal character as its name and follows
all standard spell scoping rules. The configuration block — whether a quick
form or a cauldron — is called a *codex*. This is the same codex concept
used for entity configuration (see Entity Configuration).

There are three distinct things per character and they should not be confused:

| Thing    | What it does                  | Example               |
|----------+-------------------------------+-----------------------|
| ~*~      | The glamour — applies bold    | ~*word~, ~*text*~     |
| ~:bold~  | The spell — applies bold      | ~:bold::word~         |
| ~:*~     | The config — configures ~*~   | ~:*(red)~, ~:*:::~    |

~:*~ does not apply bold. It configures what ~*~ does. ~*~ and ~:bold~ are
the action; ~:*~ is the configuration for that action.

Configuration follows all standard spell scoping rules:

#+begin_example
:*(red)                   — configure * to be red, globally from here on
:*(red)::word             — configure * to be red, just for this word
:*(red)::[this sentence]  — configure * to be red, just this scope
:*(red):: this whole line — configure * to be red, for this line
#+end_example

Global configuration (self-terminating) persists until the same spell
changes it:

#+begin_example
:*(red)
All *bold from here* is red.
More *bold still red* text.
:*(blue)
Now *bold is blue* instead.
#+end_example

The cauldron form handles multi-property codexes:

#+begin_example
:*:::
:italic
:color(red)
:grow(1)

Now all *bold text* is also italic, red, and larger automatically.
Regular /italic text/ is unaffected — only * was configured.
#+end_example

This is the same pattern used for socket configuration:

| Config target  | Quick codex         | Cauldron codex      |
|----------------+---------------------+---------------------|
| ~+~ (ordered)  | ~:+[a)]~            | ~:+:::~ + spells    |
| ~-~ (unordered)| ~:-[*]~             | ~:-:::~ + spells    |
| ~=~ (heading)  | ~:=[I.]~            | ~:=:::~ + spells    |
| ~*~ (bold)     | ~:*(red)~           | ~:*:::~ + spells    |
| ~/~ (italic)   | ~:/(blue)~          | ~:/:::~ + spells    |
| ~_~ (underline)| ~:_(dotted)~        | ~:_:::~ + spells    |

All three configuration targets — glamours, sockets, and named entities —
use the same codex system. See Entity Configuration for the full picture.

** Block styling spells

Indue is a writing tool, not a design tool. Block styling spells exist so
you do not have to leave Indue for common formatting — a highlighted callout,
a card-style container, a colored section. Anything beyond these basics is
export territory (Typst, CSS, etc.).

Block spells target a waypoint and style the containment (the box around
the content), not the text itself. Inline/cast spells target text.

#+begin_example
Block targeting (styles the container):
:bgcolor(#f5f5f5)<section>:
:radius(6)<section>:
:shadow(2,2,4)<section>:
:border(1,solid,#ccc)<section>:
:padding(1em)<section>:

Text targeting (styles the content):
:color(red)::word
:bold::italic::word
#+end_example

*** Common block spells

| Spell                       | Effect                            |
|-----------------------------+-----------------------------------|
| ~:bgcolor(value)<wp>:~      | Block background color            |
| ~:radius(value)<wp>:~       | Corner rounding                   |
| ~:shadow(args)<wp>:~        | Drop shadow on block              |
| ~:border(args)<wp>:~        | Border style                      |
| ~:padding(value)<wp>:~      | Inner spacing                     |
| ~:textshadow(args)<wp>:~    | Shadow on text (not block)        |

These compose via cauldrons when multiple properties are needed:

#+begin_example
:<section>:::
:bgcolor(#f5f5f5)
:radius(8)
:shadow(2,2,4)
:padding(1em)
#+end_example

*** Preset spells

For common patterns, higher-level spells bundle properties together:

| Spell              | Bundles                                     |
|--------------------+---------------------------------------------|
| ~:card<wp>:~       | radius, shadow, padding, border             |
| ~:chip<wp>:~       | pill radius, bgcolor, small padding         |
| ~:callout<wp>:~    | left border accent, bgcolor, padding        |
| ~:highlight<wp>:~  | bgcolor, slight radius                      |

These are convenience spells that expand into block spells. Override
individual properties after the preset:

#+begin_example
:card<section>:
:bgcolor(#fef3c7)<section>:    — override just the background
#+end_example

** Height context in layouts

Numeric height values are interpreted differently depending on the layout
context. The user types a bare number; the containing spell determines
the unit:

| Context        | Bare number means | Example                  |
|----------------+-------------------+--------------------------|
| ~:layout~      | em units          | ~:height(<wp>)(30)~ = 30em |
| ~:pagelayout~  | percentage        | ~:height(<wp>)(30)~ = 30%  |

To override the default unit, specify it explicitly:

#+begin_example
:height(<wp>)(30em)
:height(<wp>)(30%)
#+end_example

In flow layouts (~:layout~), height sets a minimum — content that exceeds
it simply grows. No overflow, no scrolling. In page layouts
(~:pagelayout~), height is a hard constraint — content that exceeds it
scrolls within its allocated space.

* List Nesting: Two Modes

Indue supports two modes for list nesting. Both produce the same AST. The
writer chooses whichever suits their context.

** Indented mode

Nesting is determined by physical indentation (tabs or spaces). The parser
normalizes both to indent levels. Tabs are recommended for proportional fonts
because they render at a visible width.

#+begin_example
- Level 1 item
  - Level 2 item
    - Level 3 item
      Content belongs to Level 3
  + Level 2 ordered item
- Another Level 1 item
#+end_example

Ownership rule: any line indented strictly deeper than a socket belongs to
that socket. The exact number of spaces does not matter as long as it is
deeper than the parent (Option C).

** Dash-mode

Nesting is determined by repeating the dash character. The final character
in the sequence determines the list type.

#+begin_example
- Level 1 unordered
-- Level 2 unordered
--- Level 3 unordered
-+ Level 2 ordered
--+ Level 3 ordered
#+end_example

Dash-mode rules:

- Count of leading dashes determines depth.
- Final character determines type: ~-~ for unordered, ~+~ for ordered.
- Non-socket lines (no leading dashes) belong to the most recent socket
  above them. Period.
- Leading whitespace on content lines is cosmetic and stripped by the parser.
- A blank line ends the list.
- Trailing child content is not supported. To return to a parent level,
  create a new socket with fewer dashes.
- Empty sockets are real empty items, not invisible or magic.

The same content in both modes:

#+begin_example
Indented:                    Dash-mode:
- Project Alpha              - Project Alpha
  - Goals                    -- Goals
    - Item A                 --- Item A
    - Item B                 --- Item B
  + Timeline                 -+ Timeline
    + Q1                     --+ Q1
    + Q2                     --+ Q2
- Project Beta               - Project Beta
#+end_example

** Mixing modes

Within a single list, one mode must be used consistently. Dash-mode and
indented mode cannot be mixed at the same nesting level. Different lists
in the same document may use different modes.

* Heading Promotion

Any list item in either mode can be promoted to a heading by appending
~= ~ (equals space) after the socket characters. The ~=~ is separated from
the socket by a space.

#+begin_example
Indented:                    Dash-mode:
- = Project Alpha            - = Project Alpha
  - = Goals                  -- = Goals
    - Item A                 --- Item A
    - Item B                 --- Item B
  + = Timeline               -+ = Timeline
    + Q1                     --+ Q1
    + Q2                     --+ Q2
- = Project Beta             - = Project Beta
#+end_example

** What heading promotion does

A heading-promoted item:

- Appears in the table of contents / outline.
- Receives an auto-calculated heading level (see below).
- Can be navigated to via the heading outline.
- Is otherwise a normal list item — it follows list ownership rules.

A non-promoted item:

- Does not appear in the ToC or outline.
- Is not assigned a heading level.
- Is just content within the list.

** Heading auto-leveling

Heading levels are not derived from list depth. They are derived from the
heading hierarchy above the current item. Each ~=~ promotion looks up the
tree for the most recent heading and adds one to its level.

#+begin_example
= Document Title                        ← H1
== Background                           ← H2
- = Project Alpha                       ← H3 (parent heading is H2)
  - Item A
  - = Goals                             ← H4 (parent heading is H3)
    - Item C
    - = Stretch Goals                   ← H5 (parent heading is H4)
  - = Timeline                          ← H4 (sibling of Goals)
- = Project Beta                        ← H3 (sibling of Alpha)
== Conclusion                           ← H2 (back to document level)
#+end_example

The writer never thinks about H-numbers. They just write ~- =~ and the
hierarchy resolves itself from context.

** Standalone headings

For flat document-mode writing (no list structure), standalone headings use
~=~ at line start with depth by repetition:

#+begin_example
= Heading 1
== Heading 2
=== Heading 3
#+end_example

Multiple ~=~ symbols only matter in flat files where all headings are at the
same physical margin. In indented structures, a single ~=~ is sufficient
because depth is calculated from ancestry.

* Folding and Outline: Two Independent Features

Indue separates physical structure from meaning structure. Folding controls
what is visible. The heading outline controls how meaning is organized.

** Standalone heading folding (legacy/document behavior)

Standalone headings (~=~, ~==~, ~===~ at the document level, outside of any
list) fold by heading hierarchy, consistent with the behavior users expect
from Org-mode, Markdown editors, and word processors. A standalone heading
folds everything until the next heading of equal or higher level.

#+begin_example
= Introduction                    ← folds until next =
Some paragraph text.
Another paragraph.

== Background                     ← folds until next == or =
More text here.

== Methods                        ← folds until next == or =
Even more here.

= Conclusion                     ← folds until next =
#+end_example

A standalone heading's fold scope includes any lists beneath it — folding the
heading hides the entire list and everything inside it, as a single unit.

** List folding (physical structure)

Inside a list, folding is controlled entirely by list structure. Every list
item with children is foldable — fold it and its children hide. This is
determined purely by indentation or dash depth. Heading promotions (~- =~)
inside lists have *zero impact* on folding. A heading-promoted list item
folds because it is a list item with children, not because it carries a
heading.

#+begin_example
= Methods                        ← standalone heading, folds by hierarchy
- = Data Collection               ← folds by LIST structure (has children)
  - Item A                        ← list child
  - Item B                        ← list child
  - = Analysis                    ← folds by LIST structure (has children)
    - Sub item                    ← list child
- = Results                       ← folds by LIST structure (has children)
  - Finding 1                     ← list child
#+end_example

In this example, folding "Methods" (a standalone heading) hides the entire
list underneath it. But within the list, folding "Data Collection" hides
only its children (Item A, Item B, and the Analysis section). The ~=~ on
"Data Collection" and "Analysis" has no effect on folding — those items fold
because they have child list items beneath them.

** Heading outline (meaning structure)

The heading outline is a separate navigational view showing only ~=~ promoted
items. It provides a table of contents and jump-to navigation. The outline
includes both standalone headings and heading-promoted list items. Headings
can be folded independently within the outline view.

The outline concerns itself with meaning: what the document is about, how
ideas relate hierarchically. It does not control physical folding.

** The handoff rule

The recommended behavior for applications:

1. Standalone headings fold by heading hierarchy (legacy behavior).
2. The moment a list begins within a heading's scope, list structure takes
   full control of folding.
3. Heading promotions inside lists are visible in the outline/ToC but are
   invisible to the folding system.
4. Folding a standalone heading hides everything inside it, including any
   lists, as a single unit.

These features never conflict because they serve different purposes. A list
fold says "hide this content." A heading outline says "show me the structure."

A document without any ~=~ headings is still fully foldable — it is lists all
the way down. A flat document with only standalone ~=~ headings still has a
navigable outline. The two features are completely orthogonal.

** Invisible markers for prose-style documents

Using a schema declaration, list markers can be made invisible:

#+begin_example
:-[]
- = Introduction
  This looks like body text under a heading.
  No visible bullet. Just indented prose.
  - = Background
    More body text.
  - = Methods
    And more here.
- = Conclusion
  Final thoughts.
#+end_example

Folds naturally from list structure. ToC comes from heading promotions. The
source looks like a conventional document but has full outliner capabilities.

* Content Ownership

** The core rule

Content belongs to the most recent socket above it. There is no trailing
child content — you cannot "return" to a parent level with content. To
place content at a parent level, create a new socket at that level.

This rule applies in both indented mode and dash-mode. It eliminates an
entire class of ambiguity that plagues Markdown, CommonMark, and Org-mode
implementations.

** The ~:of()~ escape hatch

In rare cases where content genuinely needs to attach to an ancestor, the
~:of~ spell explicitly reattaches content to a named waypoint:

#+begin_example
- = Project Alpha <alpha>
  - = Goals
    - Item A
    - Item B
  :of(<alpha>) This thought belongs to Project Alpha
#+end_example

This is rare, explicit, and uses existing spell/waypoint syntax. No new
concepts required.

** Multiline content shortcuts

Scrolls ~[]~ and conduits ~{}~ act as escape hatches for quick multiline
content that does not want to manage indentation line-by-line. The opening
bracket at the correct indentation level signals ownership:

#+begin_example
- Level 1
  - Level 2
    [belongs to
level 2 without worrying about indent]
    - Level 3 {
belongs to level 3 with *bold* and #tags}
- Level 1
#+end_example

Scrolls preserve content verbatim. Conduits parse their content with the
full Indue grammar.

* Glamours: Inline Formatting

| Syntax        | Result             | Notes                        |
|---------------+--------------------+------------------------------|
| ~*word~       | Bold               | Single-word (to whitespace)  |
| ~*text*~      | Bold               | Paired                       |
| ~*[text]~     | Bold               | Bracket form (scroll)        |
| ~*{text}~     | Bold               | Bracket form (conduit)       |
| ~*/[text]~    | Bold + italic      | Stacked glamours (bracket)   |
| ~* line~      | Bold whole line    | Line-start rule              |
| ~*/ line~     | Bold+italic line   | Line-start stacked           |

Glamours operate in four ways:

- *Line-start rule*: a glamour (or stack of glamours) at the start of a line
  followed by a space, with no closer, applies to the entire line. Stacking
  works at line start: ~*/~ makes the whole line bold and italic.
- *Single-word rule*: a glamour touching the left side of a word, with no
  closer on the line, applies until the next whitespace. Same logic as
  badges (~#tag~) and signets (~@name~).
- *Paired rule*: if a matching closer exists on the same line, the glamour
  spans everything between opener and closer.
- *Bracket rule*: a glamour (or stack) followed immediately by a scroll ~[]~
  or conduit ~{}~ applies to the bracket's contents. Multiple glamours can
  be stacked before the bracket to combine effects.

Symmetric nesting is not supported. The outer glamour wins:

#+begin_example
*_text_*     → just bold, inner _ is literal
_*text*_     → just underline, inner * is literal
*_[text]     → bold + underline (the correct way)
#+end_example

Inside a conduit, additive glamours work but redundant ones are ignored:

#+begin_example
*/{this is bold+italic with ^[superscript bit] in it}
#+end_example

The ~^[superscript bit]~ adds superscript on top of the outer bold+italic.
Writing ~/[italic]~ inside would be redundant (already italic from the outer
stack) and is silently ignored.

** Glamour-spell symmetry

Each glamour character has a corresponding named spell that does the same
thing. The glamour is shorthand for the spell+cast form:

| Glamour            | Spell equivalent              | Scope       |
|--------------------+-------------------------------+-------------|
| ~*word~            | ~:bold::word~                 | Word        |
| ~* whole line~     | ~:bold:: whole line~          | Line        |
| ~*[text]~          | ~:bold::[text]~               | Bracket     |
| ~*text*~           | ~:bold::[text]~               | Paired      |
| ~*/[text]~         | ~:bold::italic::[text]~       | Chained     |
| ~*/ whole line~    | ~:bold::italic:: whole line~  | Chained line|

The full mapping of glamour characters to spell names:

| Glamour | Spell        | Config spell |
|---------+--------------+--------------|
| ~*~     | ~:bold~      | ~:*~         |
| ~/~     | ~:italic~    | ~:/~         |
| ~_~     | ~:underline~ | ~:_~         |
| ~%~     | ~:strike~    | ~:%~         |
| ~^~     | ~:super~     | ~:^~         |

Note the three distinct roles: the glamour character applies the effect,
the named spell applies the same effect via spell syntax, and the config
spell (using the literal character) configures how the glamour behaves.
~*~ is what you type. ~:bold~ is what you chain. ~:*~ is what you configure.

This means glamour stacking (~*/[text]~) is literally cast chaining
(~:bold::italic::[text]~) expressed in shorthand. A toolset maker can
implement glamours as a preprocessing step that expands them into spell+cast
form, then run a single unified engine.

For users, this creates a natural upgrade path: start with ~*bold*~ because
it is familiar, discover ~*[bold]~ when brackets are needed, learn
~*/[bold italic]~ when stacking is needed, and when color or sizing is
required, the cast syntax is already the same pattern:
~:bold::color(red)::[text]~. No conceptual leap — just more spells in the
chain.

* Badges: Tags

Badges (tags) use ~#~, the universal tagging character:

#+begin_example
- Buy groceries #errand #personal
- Read about #machine learning# approaches
#+end_example

- Single-word badge: ~#errand~
- Multi-word badge: ~#machine learning#~ (closing ~#~ needed only for
  multi-word badges)
- Whole-line badge: ~#~ at line start badges the entire line

Badges attach semantically to the block element they appear within. Badges
and signets are similar by design — both attach meaning to a block. The only
reason they have different syntax (~#~ vs ~@~) is so that tools can
/optionally/ distinguish them. A simple renderer may treat them identically;
a PKM app might treat ~@~ as a person and ~#~ as a category. Neither
interpretation is required.

* Signets: Entity References

Signets (entity references) use ~@~:

#+begin_example
@alice mentioned this in the meeting.
See @intro for background.
#+end_example

Signets can refer to people, documents, waypoints, or any named object in the
system. Like badges, they may or may not resolve to a specific target — a
signet without a matching waypoint is still valid and carries meaning.

* Waypoints and Coordinates: Labels and Block IDs

** Waypoints (~<label>~)

Waypoints (labels) are human-readable anchors placed at the end of a heading
or block element:

#+begin_example
- = Introduction <intro>
See @intro for background.
#+end_example

A waypoint is a navigation target. Signets and badges can optionally resolve
to waypoints, but do not need to.

** Waypoint trails

Waypoints in Indue are not static, isolated anchors. When multiple waypoints
share the same name, the parser automatically networks them together to form
navigational paths through the document.

*** Undefined trails (the automatic loop)

When the parser detects multiple identical waypoints, it links them into an
undefined trail. Activating a waypoint jumps the user to the next one in
document order. When the last waypoint is reached, it cycles back to the
first, creating a navigational loop.

#+begin_example
- = Morning Routine <routine>
  Wake up, coffee, journal.

- = Afternoon Routine <routine>
  Deep work, meetings.

- = Evening Routine <routine>
  Review, wind down.
#+end_example

Navigating ~<routine>~ cycles: Morning → Afternoon → Evening → Morning.

*** Portals

A portal is an undefined trail containing exactly two waypoints. It acts as
a two-way mirror, letting the user jump back and forth between two specific
points in the text:

#+begin_example
- = Budget Summary <budget>
  See the detailed breakdown for numbers.

- = Budget Breakdown <budget>
  Detailed line items here.
#+end_example

Activating either ~<budget>~ jumps to the other. Portals are a natural
alternative to inline links for connecting related content within the same
document.

*** Defined trails (the constellation)

When you need to guide a reader through a specific, non-linear sequence,
upgrade an undefined trail by explicitly enumerating waypoints using a pipe
divider and an index:

#+begin_example
- = Conclusion <argument | 3>
  Therefore we can see that...

- = Evidence <argument | 2>
  The data shows...

- = Thesis <argument | 1>
  We propose that...
#+end_example

Despite their document order (Conclusion, Evidence, Thesis), the trail
follows the defined sequence: Thesis (1) → Evidence (2) → Conclusion (3).

Syntax: ~<name | index>~. The parser trims all whitespace around the ~|~
divider and index. ~<argument|1>~ and ~< argument | 1 >~ are identical.

*** The trail terminator (~x~)

To break the automatic cycle, replace the index with ~x~:

#+begin_example
<argument | 1>    — start
<argument | 2>    — middle
<argument | x>    — end (trail stops here, no cycling)
#+end_example

Without an ~x~ terminator, defined trails cycle like undefined ones. The
terminator makes the trail a one-way journey.

*** Resolution rules

When a signet (~@name~) or reference spell points to a waypoint name that
has multiple instances:

- *Undefined trail*: the signet resolves to the next waypoint in document
  order from the current position. This is natural for portals and
  sequential navigation.
- *Defined trail*: the signet resolves to index 1 (the start of the trail).
- *Mixed defined and undefined*: if the same name has both indexed and
  un-indexed waypoints, the defined trail is followed first in index order.
  Undefined waypoints are appended after the defined sequence in document
  order.
- *Terminated trail with orphaned undefined waypoints*: if a defined trail
  ends with ~x~, undefined waypoints with the same name are never linked
  into the trail. They become standalone anchors. This is the user's
  responsibility to manage.

** Coordinates (~<<id>>~)

Coordinates (block IDs) are opaque, machine-generated identifiers. They are
stored as a property of the block node in the AST — not as part of the text
content. They enable stable cross-references that survive renames and edits.

#+begin_example
- = Research phase <<n3b7c2>>
  - Survey competitors <<n5f1a8>>
#+end_example

Coordinates do not participate in trails. They are always unique identifiers
for a single block.

* Spells: Functions and Evaluation

** The spell prefix (~:~)

The ~:~ character is the spell (function) prefix. It replaces Typst's ~#~ for
this role, freeing ~#~ for badges.

A bare ~:~ by itself is shorthand for ~:()~ — an atomic null that renders
nothing and immediately self-terminates. This means a line like
~= : Project Title~ is safe; the bare colon terminates before the parser
reaches the title text.

** The cast operator (~::~)

~::~ is the cast — it directs a spell's effect at a target. It serves two
roles based on what follows:

- *Chaining*: if followed by another spell, the cast pushes effect forward
  to the next spell.
- *Word-cast*: if followed by a word with no space, the spell applies to
  that word only.
- *Line-cast*: if followed by a space, the spell applies to the rest of
  the line.

#+begin_example
:x::word               — word-cast: only "word" is affected
:x:: entire line       — line-cast: everything after is affected
:f1::f2::content       — chaining: f1 casts into f2, f2 casts into content
#+end_example

Every ~::~ is doing the same job: pushing effect forward. The parser sees ~::~
and looks at what follows — if it is a spell name, it chains; if it is a word
or space, it targets content.

** Spell forms

Indue deliberately keeps spell signatures simple. The design prefers more small
spells over fewer complex ones.

| Form              | Description                                       |
|-------------------+---------------------------------------------------|
| ~:fn~             | Switch — no arguments, self-terminating            |
| ~:fn:~            | Explicit self-termination (equivalent to ~:fn~)    |
| ~:fn()~           | Explicit empty vessel (equivalent to ~:fn~)        |
| ~:fn(value)~      | Single value in a vessel                           |
| ~:fn[text]~       | Verbatim text in a scroll — no parsing inside      |
| ~:fn{text}~       | Rich text in a conduit — full Indue grammar inside |
| ~:fn(val){text}~  | Vessel + conduit                                   |
| ~:fn(v,"string")~ | Vessel with value + string literal                 |

** Spell argument binding

Arguments bind to a spell when there is no whitespace between them. All
brackets and parentheses glued to the spell name are consumed by it:

#+begin_example
:fn(val)[text]{rich}   — all three bound to :fn (no whitespace)
:fn(val) [text]        — only (val) bound; [text] is separate content
#+end_example

Vessels ~()~ are always evaluated first regardless of position.

** Multiple adjacent brackets

When multiple scrolls ~[]~ or conduits ~{}~ appear adjacent to a spell (no
whitespace between them), each bracket acts as a division hint. The spell
decides how to interpret them:

#+begin_example
:table{Col1|Col2|Col3}{Val1|Val2|Val3}{Val1|Val2|Val3}
#+end_example

Each ~{}~ is a conduit. Pipes ~|~ separate cells within each conduit (the
~:table~ spell splits on them). Another spell could interpret the divisions
differently or ignore them.

** Multiline spell binding (clump rule)

After a spell, subsequent lines that begin with ~[~, ~(~, ~{~, or ~:~ (after
stripping leading whitespace) continue the spell's argument clump. A blank
line or a line starting with any other character ends the clump.

#+begin_example
:table{Col1|Col2|Col3}
{Val1|Val2|Val3}
{Val1|Val2|Val3}

This line is outside the table — blank line broke the clump.
#+end_example

This works inside lists because the parser strips indentation before checking
the first character:

#+begin_example
- Here's my data
  :table{Col1|Col2|Col3}
  {Val1|Val2|Val3}
  {Val1|Val2|Val3}
#+end_example

** The cauldron (~:::~)

When multiple spells need to cooperate on the same output, the cauldron ~:::~
opens a spell block. Subsequent lines starting with ~:~ are part of the
cauldron.

A cauldron terminates when:

- A blank line is encountered.
- A single ~:~ is encountered (explicit terminator, renders nothing).
- A non-spell line is encountered.

A cauldron can direct its combined output using ~::~ (cast) at the end:

#+begin_example
Simple cauldron (terminates at blank line):
:table:::
:border(1)
:th{Col1  | Col2  | Col3 }
:tr{Val1  | Val2  | Val3 }

Outside the cauldron.

Explicitly terminated cauldron:
:table:::
:border(1)
:th{Col1  | Col2  | Col3 }
:tr{Val1  | Val2  | Val3 }
:

Outside — the bare : ended the cauldron.

Cauldron with cast (directs output):
:color(red):::
:bold
:grow(1)
::This entire line gets the combined effect.

Cauldron cast to a bracket:
:color(red):::
:bold
::[specific words get the combined effect]
#+end_example

All spells inside the cauldron are evaluated in order, top to bottom. There
is no cascading or out-of-order configuration.

** Vessels, scrolls, and conduits: bracket semantics

- ~()~ — *Vessel*. Carries a single value to the spell. A URL, a date, a
  number. The payload is consumed by the spell.
- ~[]~ — *Scroll*. Verbatim content. The parser treats everything inside as a
  literal string. No glamours, badges, or spells are recognized. What is
  written is exactly what comes out.
- ~{}~ — *Conduit*. Rich content. The parser steps inside, runs the full Indue
  grammar, builds a sub-AST fragment, and returns it to the parent spell.
  Glamours activate, badges resolve, nested spells fire.

*Use conduits by default.* The common recommendation is to use conduits ~{}~
unless you specifically need content stripped of all meaning — code snippets,
raw data, or literal text where glamour characters should not be interpreted.
Conduits preserve the full power of Indue inside them; scrolls deliberately
suppress it.

#+begin_example
:link(https://example.com){Read *this* #tagged}  — conduit: bold and tag work
:code(rust)[fn main() { println!("hello"); }]    — scroll: content is literal
#+end_example

** Pipe convention

The pipe character ~|~ has no special meaning at the container level. It is
simply a character that appears inside scrolls or conduits like any other.

However, certain spells — particularly ~:table~ and ~:layout~ — interpret
pipes as cell or column dividers in their inputs. This is a spell convention,
not a parser rule. The spell documentation specifies whether it recognizes
pipes.

When using pipes inside a conduit, the Indue parser runs on each cell's
content independently. When using pipes inside a scroll, the content is
verbatim and pipes are just literal characters that the spell splits on.

#+begin_example
Conduit (cells are parsed — use for formatted content):
:table{*Name*|Role|Status}
{Alice|Engineer|:> Active}

Scroll (cells are verbatim — use for raw data):
:data[1.0|2.5|3.7]
[4.2|5.1|6.8]
#+end_example

** The cast as input/target divider

In any spell, everything before ~::~ is input to the spell. Everything after
~::~ is what the spell acts on. Spells choose which sides they use:

#+begin_example
:color(red)::word                     — input: red. target: word
:bold::italic::word                   — chained casts
:table{Col1|Col2}::{formatted rows}  — input: header. target: content
:link(url)::{*click here*}            — input: url. target: rich label
#+end_example

** Fallback rendering

If an app does not recognize a spell name, the parse-don't-fail philosophy
applies: the spell wrapper is ignored and the content inside the conduit
~{...}~ or scroll ~[...]~ is rendered as plain text.

* Task States

Task states are spells. They are not a separate syntax category — they follow
all the standard spell scoping rules.

** Core shorthands

| Shorthand | Category | Symbol |
|-----------+----------+--------|
| ~:o~      | Open     | ☐      |
| ~:x~      | Closed   | ☑      |
| ~:>~      | Active   | ◐      |
| ~:!~      | Blocked  | ⚠      |

** Scoping behavior

Task shorthands follow the same cast rules as any spell:

| Form                  | Effect                                       |
|-----------------------+----------------------------------------------|
| ~:x~                  | Atomic — renders the ☑ symbol only            |
| ~:x::word~            | Word-cast — strikes out one word              |
| ~:x:: rest of line~   | Line-cast — strikes out the entire line       |
| ~:x{multiple words}~  | Conduit — strikes out the enclosed content    |

** Named states

For finer-grained states beyond the four categories, named spells work the
same way:

#+begin_example
- :todo: Buy groceries
- :done: Fix the sink
- :started: Rewrite the parser
- :waiting: Hear back from contractor
- :next: File taxes
#+end_example

Named states carry category metadata:

| Named state  | Category |
|--------------+----------|
| ~:todo~      | Open     |
| ~:next~      | Open     |
| ~:started~   | Active   |
| ~:waiting~   | Blocked  |
| ~:done~      | Closed   |

Custom states can be defined and mapped to any of the four categories.

** Scheduling

Scheduling uses spells attached to task items. See the Scheduling and
Recurrence section for the comprehensive syntax including recurrence
patterns, ordinal dates, exceptions, and completion modes.

#+begin_example
- :o Buy groceries :due(2026-04-15)
- :o Weekly review :on(2026-03-10) :repeat[weekly]
- :o Book club :on(2026-04-20) :repeat[3rd saturday]
- :started: Rewrite parser :due(2026-04-01)
#+end_example

** Interaction with sockets

Sockets are parsed before spells. Task spells apply to the content /after/
the socket:

#+begin_example
- = :x:: Whole Heading        — heading slot, title struck out
- :x::Whole Heading           — list slot, only "Whole" struck out
:x:: - = Text                 — NOT a socket; - = is literal text
#+end_example

* List and Heading Schema Declarations

List and heading marker styles are declared on standalone lines using the ~:+~,
~:-~, and ~:=~ spell operators. This avoids "indentation explosion" from
cramming metadata onto the first content line.

#+begin_example
:+[a)]          — ordered lists at this level use a), b), c)
:-[*]           — unordered lists at this level use *
:=[I.]          — headings at this level use I., II., III.
:-[]            — invisible markers (no visible bullet)
#+end_example

Declarations can be grouped using a conduit:

#+begin_example
:{ :=[I.] :+[a)] :-[*] }
#+end_example

** Scope

A declaration binds to the immediate next sibling and its entire subtree. It
"evaporates" (resets to defaults) upon a physical outdent past the declaration's
starting column.

#+begin_example
:{ :=[I.] :+[a)] :-[*] }
+ = Meeting Notes                     — renders as: I. Meeting Notes
  :=[A.]
  + = Action Items                    — renders as: A. Action Items
  + = General Notes                   — renders as: B. General Notes
    - = Ticket Audits                 — renders as: * Ticket Audits
      - Common Scenario               — renders as:   * Common Scenario
    - Uncommon Scenario               — renders as: * Uncommon Scenario
+ = Other Topics                      — renders as: II. Other Topics
#+end_example

** Pattern matching

The template inside the scroll ~[...]~ is a literal example of the first
marker. The parser uses it to auto-increment subsequent items. Users may also
type the number/letter themselves for human readability — the parser treats the
typed marker as a "hint" and corrects the sequence if needed.

Supported patterns include ~[1.]~, ~[a)]~, ~[(1)]~, ~[I.]~, ~[A.]~, and
similar alphabetic/numeric/roman variations.

** User-typed ordered markers

If a schema is declared (e.g. ~:+[a)]~), the user may also type the markers
directly:

#+begin_example
:+[a)]
a) hello
b) what's up
c) yeah
#+end_example

The parser matches these against the declared pattern at line start.

* Block-Level Content

** Tables

Tables use the ~:table~ spell. The ~:table~ spell interprets pipes ~|~ as
cell dividers in its inputs. Use conduits by default so cell content can
include glamours, badges, and spells. Use scrolls for raw data where no
parsing is needed.

#+begin_example
Simple table (conduits, cells are parsed):
:table{Col1|Col2|Col3}
{*Alice*|Engineer|:> Active}
{Bob|Designer|:o Open}

Raw data table (scrolls, cells are verbatim):
:data[x|y|z]
[1.0|2.5|3.7]
[4.2|5.1|6.8]

Configured table (cauldron):
:table:::
:border(1)
:th{Col1  | Col2  | Col3 }
:tr{*Alice*  | Engineer  | :> Active }
:tr{Bob  | Designer  | :o Open }
#+end_example

The simple form uses adjacent conduits — the spell infers rows from each
conduit and columns from pipes. The cauldron form allows configuration spells
like ~:border~ and distinguishes headers (~:th~) from rows (~:tr~).

** Layout

The ~:layout~ spell arranges content blocks into columns. Content is written
linearly with waypoints; the layout spell maps those waypoints into a spatial
arrangement. In plain text you read sections top to bottom. In a renderer
they flow into their assigned positions.

Layout handles horizontal columns only. Vertical stacking is natural content
flow. If one column has more content than another, the shorter columns divide
their space equally among their waypoints.

#+begin_example
:layout{<analysis>|<numbers>}:

- = Market Analysis <analysis>
  The market is growing rapidly.
  Revenue increased 40% year over year.
  Key competitors include several startups.

- = Financial Summary <numbers>
  Q1: $2.3M
  Q2: $2.8M
  Q3: $3.1M
  Q4: projected $3.5M
#+end_example

The conduit is used because ~<waypoints>~ are parsed Indue syntax. Pipes
divide columns. Adjacent waypoints within a column stack vertically:

#+begin_example
:layout{<nav>|<content>}:                — 2 columns
:layout{<a>|<b>|<c>}:                   — 3 columns
:layout{<top><bottom>|<side>}:          — left col split, right col full
:layout{<a><b>|<c><d>}:                 — 2x2 grid
#+end_example

Column sizing targets the waypoint directly:

#+begin_example
:width(<nav>)[30]
:layout{<nav>|<content>}:
#+end_example

The ~:width~ spell sets a percentage. Columns without a width declaration
share the remainder equally. The last column never needs a width.

For more control, use a cauldron:

#+begin_example
:layout:::
:width(<nav>)[30]
:overflow(<content>)[scroll]
:gap[1em]
{<nav>|<content>}
#+end_example

** Page-constrained layout (~:pagelayout~)

For dashboards, slides, and reports where content must fit within a defined
page boundary, use ~:pagelayout~ instead of ~:layout~. Heights are
percentages of the page. Content that exceeds its allocation scrolls.

#+begin_example
:pagelayout:::
:height(<summary>)(40)
:height(<details>)(60)
{<summary><details>|<sidebar>}
#+end_example

| Feature    | ~:layout~ (flow)      | ~:pagelayout~ (constrained)   |
|------------+-----------------------+-------------------------------|
| Height     | Content-driven (em)   | Percentage of page            |
| Overflow   | Never — grows to fit  | Scrolls                       |
| Page break | Content spans pages   | Confined to current page      |
| Use case   | Documents, notes      | Dashboards, slides, reports   |

** Page configuration (~:page~)

~:page~ configures the page itself — size, margins, headers, footers. This
is document-level setup, not content layout. Configuration persists until
changed.

#+begin_example
:page(a4):
:page(letter):
:page:::
:orientation(landscape)
:margin(2cm)
:header{Document Title}:
:footer{Page :pagenum:}:
:columns(2)
#+end_example

** Page and element spells

#+begin_example
:break(page):                  — force a new page
:align(center)<wp>:            — center a waypoint's content
#+end_example

Deleting any layout or page line causes content to render linearly — nothing
breaks. Layout is a rendering instruction, not a structural dependency.

** Code blocks

Code blocks use the ~:code~ spell:

#+begin_example
:code(rust)
[fn main() {
    println!("hello");
}]
#+end_example

The scroll preserves content verbatim with no internal parsing.

** Reference and transclusion spells

| Spell              | Behavior                                 |
|--------------------+------------------------------------------|
| ~@name~            | Signet — summons entity by name          |
| ~:link(url){t}~    | Navigation — click to go there           |
| ~:of(<wp>)~        | Reattach content to ancestor waypoint    |
| ~:embed(<wp>):~    | Transclusion — pulls content inline      |
| ~:popover(<wp>):~  | Popup — hover/click to show content      |
| ~:query(expr)~     | Live query — filters matching blocks     |

Signets, waypoints, coordinates, and badges are all different addressing modes
for the same underlying reference system. All reference spells accept any of
them.

*** Transclusion (~:embed~)

Transclusion pulls referenced content inline at the point where the spell
appears. The source content appears as if it were written here.

#+begin_example
:embed(<intro>):                — transclude block with waypoint <intro>
:embed(<<n3b7c2>>):            — transclude by coordinate
:embed(#work):                 — transclude all blocks badged #work
:embed(@alice):                — transclude entity content
#+end_example

Whether transclusion is one-way (read-only mirror) or two-way (editable in
place) is an application decision, not a language requirement. The spec
defines only that the referenced content appears at the transclusion point.

*** Popover (~:popover~)

Popovers display referenced content in a floating panel on hover or click,
without leaving the current context. An optional scroll provides a display
label:

#+begin_example
:popover(<wp>):                — shows block content on hover/click
:popover(<wp>)[Details]:       — with custom label
:popover(@alice):              — entity info card
:popover(:query(#work :>)):    — query results in a panel
:popover[Tooltip text]:        — static text, no reference needed
#+end_example

*** Chaining references

Reference spells chain with casts like any other spell:

#+begin_example
:link(url)::embed(<wp>):       — link with inline transclusion
:link::popover(<wp>):          — link with hover preview
:link(url)::{*click here*}     — link with formatted label
#+end_example

* Entity Configuration: The Indue Spell

The namesake feature of the language. The ~:indue~ spell — and its shorthand
forms — configures named entities (waypoints, badges, signets) so that any
block where the entity appears automatically inherits the configured spells.
This is Indue's answer to "supertags": a badge like ~#project~ can carry
formatting, task defaults, or any other spell to every block it touches.

** The core metaphor

Plain text puts on Indue and gains power. The ~:indue~ spell defines /what/
that power is for a given entity name. Every block that wears the entity
inherits its properties — indued with the qualities it did not previously
have.

** Syntax

The ~:indue~ spell follows the standard configuration pattern. There are
three levels:

#+begin_example
Type configuration (all waypoints, all badges, etc.):
:<>:::              — configure how all waypoints look/behave
:#:::               — configure how all badges look/behave
:@:::               — configure how all signets look/behave

Entity configuration (specific named entity):
:<project>:::       — configure the "project" entity (shorthand)
:indue(<project>):::— same thing (longhand)

Inside the cauldron, standard spells define the inherited properties:
:<project>:::
:bgcolor(#f0f0f0)
:border(1, solid, #ccc)
:padding(0.5em)
#+end_example

Any block that carries ~<project>~, ~#project~, or ~@project~ automatically
inherits the configured bgcolor, border, and padding. The entity name is the
identity — the sigil type (~#~, ~@~, ~<>~) is just how you reference it.

** Codex

A *codex* is any configuration block in Indue — the set of spells that
define how a glamour, socket, or named entity behaves. The term applies
uniformly across the entire configuration system:

| Codex target       | Quick codex      | Cauldron codex              |
|--------------------+------------------+-----------------------------|
| ~*~ (glamour)      | ~:*(red)~        | ~:*:::~ + spells            |
| ~+~ (socket)       | ~:+[a)]~         | ~:+:::~ + spells            |
| ~<project>~ (entity)| —               | ~:<project>:::~ + spells    |
| ~<>~ (type)        | —                | ~:<>:::~ + spells           |

A glamour codex configures how a formatting character renders. A socket
codex configures how a list or heading marker renders. An entity codex
configures inherited properties that are bestowed on every block where
that entity appears. All three use the same cauldron syntax and the same
scoping rules.

#+begin_example
:<meeting>:::
:bgcolor(#e8f4fd)
:callout
:+[1.]

This codex makes every block tagged #meeting or anchored <meeting>
automatically appear as a numbered callout with a light blue background.
#+end_example

** Scope and persistence

Codex effects are applied to the current block and forward. If a codex
appears in the middle of a document, blocks above it are not affected.

#+begin_example
- Regular block #project             — no codex yet, default styling

:<project>:::
:bgcolor(#f0f0f0)

- Styled block #project              — inherits bgcolor from codex
- Another styled block #project      — also inherits
#+end_example

A later codex for the same entity overrides the previous one:

#+begin_example
:<project>:::
:bgcolor(#f0f0f0)

- Block A #project                   — light gray

:<project>:::
:bgcolor(#ffffcc)

- Block B #project                   — light yellow (overridden)
#+end_example

** The satchel (~)

To hoist configuration to the top of the document (like HTML's ~<head>~ or
a CSS style block), use a *satchel*. A satchel is a block delimited by ~
characters that is evaluated before the body text, regardless of its
physical position in the file.

*** Satchel syntax

Any sequence of one or more ~~ characters acts as a satchel delimiter.
Multiple ~~ characters in a row are treated as a single delimiter —
~\~\~\~\~\~~ is the same as ~~~. This allows visual decoration:

#+begin_example
~~~~~~~~~~
:<project>:::
:bgcolor(#f0f0f0)
:border(1)

:<meeting>:::
:bgcolor(#e8f4fd)
:callout
~~~~~~~~~~

= Project Alpha <project>
Content here automatically gets the project styling.

- Team sync #meeting
  This gets the meeting styling.
#+end_example

*** Header, middle, and footer satchels

Satchels can appear anywhere in the document. Common patterns:

#+begin_example
Header satchel (best practice — clean start):
~~~~~~~~~~
Configuration here
~~~~~~~~~~
= Title
Content begins immediately.

Middle satchel (configuration change mid-document):
= Part One
Content...
~~~~~
New configuration
~~~~~
= Part Two
Content with updated config...

Footer satchel (no closing needed — terminates at end of file):
= Content
More content...
~~~~~~~~~~
Configuration here — no closing ~ required at end of document.
#+end_example

The footer form is especially useful: type ~~~~~~~~~~  as a visual divider
near the bottom and put all your codexes below it. Best practice is to keep
the top of the document clean, going straight into a title or heading, and
putting configuration in a footer satchel.

*** Satchel evaluation order

1. Applications scan all files in the vault scope and auto-import
   configurations.
2. Satchel contents from all files are hoisted and evaluated first.
3. The body is then parsed with full knowledge of all codexes.
4. Inline codexes (not in satchels) override hoisted ones when encountered
   in document order.

This means a codex defined in one file's satchel can affect blocks in
another file — the vault is a single configuration namespace. Application
developers should document their vault scanning behavior.

** Codex for waypoints

A waypoint codex defines how a specific waypoint and its associated entity
affect the blocks where they appear. You can place a waypoint above its
codex to make the codex itself navigable:

#+begin_example
<project>
:<project>:::
:bgcolor(#f0f0f0)
:card
#+end_example

Now ~@project~ in a signet jumps to the codex definition. The codex is its
own documentation.

** Interaction with configuration hierarchy

All codexes compose in a predictable order:

- *Glamour/socket codex* (~:*:::~, ~:+[a)]~) affects how a character
  renders globally.
- *Entity codex* (~:<project>:::~) affects blocks where that entity appears.
- *Inline spells* (~:bgcolor(red)::~) affect specific content in place.

When multiple codexes apply, inline spells override entity codexes, which
override glamour/socket codexes. Explicit always wins over inherited.

** Open questions for entity configuration

- Should the codex affect blocks that reference the entity via ANY form
  (~#name~, ~@name~, ~<name>~), or only when a waypoint is physically
  present? Current design: all forms share the entity namespace and the
  codex applies to all.
- Should codex properties be composable? If a block has ~#project~ and
  ~#urgent~, do both codexes apply? If so, what order? (Likely: applied in
  the order the entities appear on the block, last wins for conflicts.)
- Should there be a way to define a codex that only applies to blocks with
  a specific combination of entities (intersection)?

* Parser Architecture Notes

** Three-layer grammar

The formal grammar should be defined in three distinct layers:

1. *Socket layer*: identifies list items, heading promotions, and
   coordinates. This is line-start based. Sockets are parsed first.
2. *Glamour layer*: identifies inline formatting (bold, italic, underline,
   strikethrough, superscript). These follow the line-start, single-word,
   paired, and bracket rules. Stacking and symmetric nesting rules apply.
3. *Entity layer*: identifies signets (~@~), spells (~:~), badges (~#~), and
   bracket content (vessels, scrolls, and conduits).

** Indentation handling

List nesting and content ownership in indented structures are determined by
physical indentation. Both tabs and spaces are accepted; the parser normalizes
to indent levels. The specific number of spaces per level does not need to be
consistent — only the relative depth matters (Option C: strictly deeper than
the parent socket).

** Heading level logic

Heading levels are derived from the heading hierarchy, not from list depth.
Each ~=~ promotion looks up the tree for the most recent heading and adds one.
In flat document mode, the count of ~=~ symbols determines level directly.

** Stateful list parsing

When the parser encounters a schema declaration (~:+[a)]~), it updates a style
dictionary keyed by indentation depth and list type. When the indentation
outdents past the declaration's column, that entry is popped from the stack.

** Coordinates in the AST

~<<id>>~ (coordinate) values are stored as a property of the block node, not
as a child of the text content. This allows apps to look up ~node.id~ directly
without traversing the text.

** Error recovery

- Unclosed glamours apply to the single word they touch, or render as
  literal text if not adjacent to a word.
- Unknown spell names render their conduit/scroll content as plain text.
- The parser never throws errors to the user.
- Strict validation belongs in the linter and export pipeline.

* Relationship to Typst

| Feature         | Typst                 | Indue                |
|-----------------+-----------------------+----------------------|
| Function prefix | ~#~                   | ~:~ (spell)          |
| Tag syntax      | ~#tag("work")~ (func) | ~#work~ (badge)      |
| Headings        | ~=~ / ~==~ / ~===~    | ~=~ / ~==~ / ~===~   |
| Bold            | ~*text*~              | ~*text*~ (same)      |
| Italic          | ~_text_~              | ~/text/~ (glamour)   |
| Underline       | None built-in         | ~_text_~ (glamour)   |
| Strikethrough   | None built-in         | ~%text%~ (glamour)   |
| Checkboxes      | None (package/show)   | ~:o~ ~:x~ (spell)    |
| Task states     | Convention            | Native spells        |
| Lists           | ~-~ / ~+~             | ~-~ / ~+~ (same)     |
| Labels          | ~<label>~             | ~<label>~ (waypoint) |
| References      | ~@ref~                | ~@ref~ (signet)      |
| Error handling  | Strict compiler       | Parse, don't fail    |

** Transpilation to Typst

- ~=~ headings pass through (same syntax).
- ~:func(...)~ (spell) becomes ~#func(...)~.
- ~#tag~ (badge) becomes ~#tag("tag")~ (via a Typst package).
- ~:o~ / ~:x~ become ~#todo~ / ~#done~ (via a Typst package).
- Bold and lists pass through unchanged.
- ~/italic/~ becomes ~_italic_~.
- ~_underline_~ becomes ~#underline[...]~.

* Scheduling and Recurrence

Indue provides a comprehensive scheduling system through spells. Scheduling
is critical to the PKM and task management use case — it must be part of the
core spec, not an afterthought.

** Date spells

| Spell                          | Purpose                        |
|--------------------------------+--------------------------------|
| ~:due(date)~                   | Task deadline                  |
| ~:on(date)~                    | Event/calendar date            |
| ~:scheduled(date)~             | Planned start date             |
| ~:start(date)~                 | Date range start               |
| ~:end(date)~                   | Date range end                 |

Dates use ISO 8601 format by default: ~2026-04-15~, ~2026-04-15T14:00~.
Natural-language dates are an application feature, not a spec requirement.

** Recurrence syntax

Recurrence is specified via the ~:repeat~ spell. The design draws from
iCalendar RRULE concepts but uses a human-readable syntax rather than the
RRULE key-value format. Recurrence patterns use scrolls because the pattern
is a literal string — no Indue syntax needs to be parsed inside.

The anchor date for recurrence comes from the sibling date spell (~:on~,
~:due~, or ~:scheduled~) on the same item. If no date spell is present,
the application may use today as the implicit anchor.

*** Simple intervals

#+begin_example
:repeat[daily]
:repeat[weekly]
:repeat[monthly]
:repeat[yearly]
:repeat[3 days]                 — every 3 days
:repeat[2 weeks]                — every 2 weeks
:repeat[6 months]               — every 6 months
#+end_example

*** Day-of-week patterns

#+begin_example
:repeat[weekdays]               — Monday through Friday
:repeat[weekends]               — Saturday and Sunday
:repeat[mon, wed, fri]          — specific days
:repeat[tue, thu]               — specific days
#+end_example

*** Ordinal patterns (irregular dates)

This is where most systems fail. Indue uses a readable ordinal syntax:

#+begin_example
:repeat[1st friday]             — first Friday of every month
:repeat[last saturday]          — last Saturday of every month
:repeat[3rd tuesday]            — third Tuesday of every month
:repeat[2nd weekday]            — second weekday of every month
:repeat[last day]               — last day of every month
:repeat[1st monday of march]    — first Monday of March (yearly)
:repeat[last friday of quarter] — last Friday of each quarter
#+end_example

Ordinal keywords: ~1st~, ~2nd~, ~3rd~, ~4th~, ~5th~, ~last~.

Day keywords: ~monday~ / ~mon~, ~tuesday~ / ~tue~, ~wednesday~ / ~wed~,
~thursday~ / ~thu~, ~friday~ / ~fri~, ~saturday~ / ~sat~, ~sunday~ / ~sun~,
~day~, ~weekday~, ~weekend-day~.

Period keywords: ~month~ (default), ~quarter~, ~year~, and specific month
names (~january~ / ~jan~ through ~december~ / ~dec~).

*** Interval + ordinal combinations

#+begin_example
:repeat[1st friday of 2 months] — first Friday, every other month
:repeat[last day of 3 months]   — last day, every 3 months
#+end_example

*** Bounds

Recurrence start and end are controlled by the existing date spells on the
same item:

#+begin_example
- :o Team meeting :on(2026-04-07) :repeat[weekly] :end(2026-12-31)
- :o Quarterly review :on(2026-01-15) :repeat[last friday of quarter]
- :o Sprint planning :scheduled(2026-04-01) :repeat[2 weeks]
#+end_example

~:end~ terminates the recurrence. Without an ~:end~, the recurrence
continues indefinitely. ~:on~ or ~:scheduled~ provides the anchor date from
which the pattern begins.

*** Exceptions

Specific occurrences can be skipped:

#+begin_example
:skip(2026-07-04)               — skip this date
:skip(2026-12-25, 2026-12-26)   — skip multiple dates
#+end_example

*** Completion behavior

Two modes for how the next date is calculated after completion:

#+begin_example
:repeat[weekly]                 — next date from the scheduled date
:repeat![weekly]                — next date from the completion date
#+end_example

~:repeat~ schedules from the original anchor (task recurs on the same day
each week regardless of when you complete it). ~:repeat!~ schedules from
when you actually finished (useful for habits where regularity from last
completion matters).

*** Full examples

#+begin_example
- :o Pay rent :due(2026-04-01) :repeat[monthly]
- :o Team standup :on(2026-04-07T09:00) :repeat[weekdays]
- :o Haircut :scheduled(2026-04-15) :repeat![6 weeks]
- :o Book club :on(2026-04-20) :repeat[3rd saturday]
- :o Tax prep :due(2026-03-15) :repeat[yearly]
- :o Sprint review :on(2026-04-12) :repeat[last friday of 2 weeks]
- :o Board meeting :on(2026-01-20) :repeat[3rd monday of quarter]
#+end_example

*** RRULE interoperability

For CalDAV/iCalendar export, each ~:repeat~ pattern has a deterministic
mapping to an RRULE string. Applications that export to calendar systems
should perform this conversion. The Indue syntax is the authoring format;
RRULE is the interchange format.

* Query System

Indue defines a minimal portable query system. The goal is that basic
queries produce consistent results across compliant applications. Advanced
querying is left to application developers.

** Design principles

- Queries are spells — they follow all standard spell rules.
- Query results are AST fragments, the same as transclusion (~:embed~).
- By default, matching blocks are inserted inline, like ~:embed~.
- Applications may extend the query language but must support the core
  syntax.

** Core query syntax

The ~:query~ spell accepts a filter expression:

#+begin_example
:query(#work):                  — all blocks badged #work
:query(:o):                     — all open tasks
:query(:x):                     — all closed tasks
:query(:>):                     — all active tasks
:query(:!):                     — all blocked tasks
:query(@alice):                 — all blocks mentioning @alice
:query(#work :o):               — open tasks badged #work (AND)
:query(#work | #personal):      — badged work OR personal
:query(:due < 2026-04-15):      — due before April 15
:query(:due = today):           — due today
:query(:due < +7d):             — due within the next 7 days
#+end_example

*** Filter operators

| Operator | Meaning             | Example                     |
|----------+---------------------+-----------------------------|
| (space)  | AND                 | ~#work :o~ = work AND open  |
| ~\vert~  | OR                  | ~#work \vert #personal~     |
| ~<~      | Before / less than  | ~:due < 2026-04-15~         |
| ~>~      | After / greater than| ~:due > 2026-01-01~         |
| ~=~      | Equals              | ~:due = today~              |
| ~!~      | NOT                 | ~!#archived~                |

*** Relative dates in queries

| Token    | Meaning            |
|----------+--------------------|
| ~today~  | Current date       |
| ~+7d~    | 7 days from now    |
| ~-3d~    | 3 days ago         |
| ~+1w~    | 1 week from now    |
| ~+1m~    | 1 month from now   |

*** Query display

By default, query results render as a flow of matching blocks (like
~:embed~). Applications may provide alternative views:

#+begin_example
:query(:o #work):                       — inline block list (default)
:query(:o #work)::table:                — render as table
:popover(:query(#work :>)):             — show in popover panel
#+end_example

*** Scope

Queries search the current document by default. Applications may extend
scope to the entire vault, specific files, or across linked documents.
The spec does not mandate vault-wide search — that is an application
feature built on top of the core query syntax.

* Compliance Tiers

Indue is designed to be implementable at different levels of capability.
Not every application needs to support every feature. The compliance tier
system ensures that less ambitious apps can still participate in the Indue
ecosystem while encouraging adoption of the features that make Indue
distinct.

** Partial compliance (recognized, not full Indue)

Applications that implement basic sockets, glamours, badges, signets,
scrolls, conduits, and parse-don't-fail error recovery — but not tasks,
scheduling, or structural features — are recognized as "Indue-aware" but
are not considered compliant Indue implementations. This level exists as
a stepping stone, not a destination. Many Markdown renderers could reach
this level with minimal effort, but it does not represent what makes Indue
distinct.

** Tier 1: Core (required)

The minimum for calling something an "Indue-compatible" application.

- Sockets: ~-~, ~+~, ~=~ and indentation-based nesting
- Glamours: all five (~*~/~_~%~^~) with line-start, single-word, paired,
  and bracket rules
- Badges: ~#tag~
- Signets: ~@ref~
- Waypoints: ~<label>~ and coordinates ~<<id>>~
- Scrolls and conduits: ~[...]~ and ~{...}~
- Basic spells: ~:fn~, ~:fn()~, ~:fn[]~, ~:fn{}~
- Cast: ~::~ for word and line scope
- Nick: ~\~ escaping
- Parse-don't-fail error recovery
- Task state spells: ~:o~, ~:x~, ~:>~, ~:!~
- Named task states: ~:todo~, ~:done~, ~:started~, ~:waiting~, ~:next~
- Scheduling: ~:due~, ~:on~, ~:scheduled~
- Basic recurrence: ~:repeat~ with simple intervals (~daily~, ~weekly~,
  ~monthly~, ~2 weeks~)
- Heading promotion: ~- =~ / ~+ =~
- Schema declarations: ~:+~, ~:-~, ~:=~
- Tables: ~:table~ with conduit rows and pipe columns
- Layout: ~:layout~ for basic column arrangements
- Link spell: ~:link~

** Tier 2: Knowledge (recommended for PKM)

Features that enable the knowledge management use case.

- Transclusion: ~:embed~
- Popovers: ~:popover~
- Basic queries: ~:query~ with badge, signet, and task state filters
- Spell chaining: ~::~ as chain operator
- Cauldron: ~:::~ for multi-spell cooperation
- Advanced recurrence: ordinal patterns, exceptions, completion modes
- Dash-mode lists

** Tier 3: Presentation (optional)

Formatting and layout features for polished output.

- Formatting spells: ~:color~, ~:bgcolor~, ~:font~, ~:grow~, ~:shrink~
- Block styling: ~:radius~, ~:shadow~, ~:border~, ~:padding~
- Preset spells: ~:card~, ~:chip~, ~:callout~
- Page-constrained layout: ~:pagelayout~
- Page configuration: ~:page~, ~:break~
- Sigil configuration: ~:*~, ~:/~, etc.

** Tier labeling

Applications should declare their compliance tier. A Tier 2 app implements
everything in Tier 1 and Tier 2. Tiers are cumulative.

The tier system explicitly discourages building Indue apps that only handle
basic formatting. There are plenty of Markdown renderers for that. Indue
exists because tasks, scheduling, and linking/graphs deserve first-class
treatment. Tier 1 apps that never progress to Tier 2 are missing the point.

* Custom Spell Extensions

Applications may define their own spells beyond those in the core spec.
Custom spells must follow all standard spell syntax rules — they use the
same ~:name~ prefix, the same vessel/scroll/conduit arguments, and the same
cast/chaining behavior.

** The ~:ext~ namespace (recommended)

To avoid name collisions between applications, custom spells should use a
namespace prefix:

#+begin_example
:ext.myapp.kanban{...}          — app-specific kanban board
:ext.myapp.timeline{...}        — app-specific timeline view
:ext.obsidian.dataview{...}     — Obsidian-specific dataview
#+end_example

Applications may also register short names that omit the namespace for
convenience within their own ecosystem:

#+begin_example
:kanban{...}                    — works in MyApp, unknown elsewhere
#+end_example

** Graceful degradation

When an application encounters an unknown spell — whether from another app's
extension or a future version of the spec — the parse-don't-fail philosophy
applies:

- The spell name is preserved in the AST as an unknown spell node.
- Content inside conduits ~{...}~ is rendered as formatted text.
- Content inside scrolls ~[...]~ is rendered as plain text.
- Vessel values ~(...)~ are preserved but not interpreted.
- Applications should visually indicate that a spell is unrecognized (e.g.
  subtle highlight, hover tooltip showing the spell name).

This means an Indue file with app-specific spells is always readable in
any other Indue application. The custom features degrade to visible content
rather than disappearing or causing errors.

** Custom spell discovery

Applications may include a spell registry in their vault configuration
(e.g. the ~.indue~ vault marker file) that declares custom spells with
descriptions, so other applications can provide better fallback rendering
or migration tools.

* Open Design Questions

** Math notation

~$~ is available and unused. Should Indue keep Typst's ~$...$~ math syntax
or define its own subset?

** Term/definition lists

~/~ is used for italic, so Typst's ~/ term: definition~ is not available.
Options include a spell like ~:def(term){definition}~ or a different sigil.

** Blockquotes

~>~ is reserved and likely for blockquotes (Markdown convention).

** Typesetting scope

Should Indue define its own page layout, or always delegate to Typst?

** Metadata and frontmatter

How should document-level metadata (title, author, date) be represented?

** Scripting and logic

Should Indue have scripting, or should logic live in host tooling?

** Named spell arguments

Currently positional only. Named arguments may be added if needed.

** Escaping edge cases

The nick (~\~) serves as a general escape character (e.g. ~\*~ for a literal
asterisk, ~\#~ for a literal hash). The full set of characters that need
escaping in various contexts should be specified precisely. In particular:
which characters need escaping inside scrolls (probably none, since scrolls
are verbatim), inside conduits, and in plain text at line start vs mid-line.

** Code block syntax

Should Indue also support backtick fences (```) alongside the ~:code~ spell
for familiarity with CommonMark conventions?

* Implementation Requirements

Building Indue as a real language requires:

- A formal grammar specification (likely PEG or context-free), defined in
  three layers: socket, glamour, and entity.
- A Rust reference CLI as the canonical implementation.
- Error handling with clear diagnostics (for the linter/export pipeline;
  the core parser follows parse-don't-fail).
- A tree-sitter grammar for editor syntax highlighting and structural queries.
- A transpiler to Typst for typeset PDF export.
- Unicode handling (UTF-8 throughout).
- A comprehensive test suite covering nesting, ambiguity, edge cases, and
  round-trip fidelity.
- Documentation of every syntactic interaction and edge case.
- A Flutter/Dart application (OUST, name pending) consuming the Rust
  implementation via Flutter Rust Bridge, targeting MechanixOS, Android,
  iOS, and desktop Linux.
